# Calculator Pricing Admin Plan (Issue #95, этап 1)

## Цель этапа
Подготовить безопасную структуру групп и метаданных для будущей админки тарифов без изменения текущей бизнес-логики расчёта.

## Текущие тарифные блоки в калькуляторе
- `finishMaterialPerM2` — ставка материалов отделки/ступеней за м².
- `railingPerM` — ставка ограждения за погонный метр.
- `lightingPerStep` — ставка подсветки за ступень.
- `coatingPerM2` — ставка покрытия за м².
- `service` — сервисные и монтажные ставки (`fitCheck`, `prepPerM2`, `installPerM2`, `fullCladdingPerM2`).
- `pricing regions` — региональные коэффициенты (`stair_pricing_regions`).
- `coefficients` — базовые коэффициенты из `pricing_defaults` (`install_coef`, `markup_coef` и т.д.).

## Группы для будущей админки
1. `step_materials` — Материалы ступеней.
2. `cladding_materials` — Отделка / облицовка.
3. `cladding_scope` — Объём отделки.
4. `finish_treatments` — Финишная обработка.
5. `railings` — Ограждения.
6. `lighting` — Подсветка.
7. `work_rates` — Работы и подготовка.
8. `pricing_regions` — Регионы.
9. `pricing_coefficients` — Коэффициенты.
10. `pricing_disclaimers` — Тексты и дисклеймеры.

## Правильное разделение материалов
- **Материалы ступеней**: дуб, ясень, бук, лиственница, карагач, термоясень.
- **Отделка / облицовка**: дубовый шпон, ясеневый шпон, МДФ под эмаль, МДФ шпон, керамогранит, камень, микроцемент.
- **Финишная обработка**: морилка, лак, масло, масло-воск, эмаль, эмаль по RAL, подбор цвета.

## Метаданные на каждый тарифный элемент
- `group_key`
- `item_key`
- `title`
- `description`
- `unit`
- `price`
- `applies_to`
- `excludes`
- `sort_order`
- `is_active`
- `client_hint`
- `admin_note`

Эти поля позволяют явно хранить:
- на что влияет тариф (`applies_to`);
- где не используется (`excludes`);
- единицу расчёта (`unit`);
- подсказку клиенту и факт показа (`client_hint`);
- мягкое отключение без удаления (`is_active`).

## Варианты внедрения

### Вариант A — единая структура `calculator_price_items`
Одна таблица/структура с `group_key` + единым набором полей.

**Плюсы (для этапа 1):**
- минимальный риск для существующей логики;
- простое поэтапное заполнение групп;
- можно параллельно маппить текущие `rate_group/rate_key` без миграции расчётчика.

**Минусы:**
- часть валидаций будет прикладной (в коде), а не строго реляционной.

### Вариант B — отдельные таблицы по каждому домену
`step_materials`, `cladding_materials`, `finish_treatments`, `railing_options`, `lighting_options`, `work_rates`, `pricing_regions`, `pricing_coefficients`, `calculator_disclaimers`.

**Плюсы:**
- строгая типизация доменов на уровне БД.

**Минусы для первого шага:**
- больше миграций и точек отказа;
- выше риск задеть рабочий расчёт и админские загрузчики.

## Выбор безопасного первого шага
Для текущего проекта безопаснее **Вариант A**: сначала унифицировать структуру метаданных, не меняя формулы и источники действующих тарифов, затем постепенно переводить UI/CRUD и только после этого решать вопрос декомпозиции на отдельные таблицы.

## Что сделано в этом PR
- Зафиксирована структура групп и полей.
- Уточнены подписи, чтобы не смешивать дуб со шпоном в текущих данных интерфейса (`Дуб` отдельно).
- Расчётные формулы и сценарии (`empty_opening`, `ready_base`, `existing_metal_frame`, `ready_frame`, `existing_concrete_base`) не менялись.

## Подготовительный этап UI: только `finish_treatments`
- В админке на этом шаге включается только read-only/read-edit представление группы `finish_treatments`.
- Сохранение, CRUD и подключение к Supabase не выполняются.
- Цена для вариантов отображается как `Не задано` (без `0`).

### Правила RAL внутри финишной обработки
- `Эмаль по RAL` — это вариант внутри `Финишной обработки`, а не отдельная обязательная группа.
- `RAL` является **необязательной** опцией: цвет можно указать сразу или согласовать позже.
- Отсутствие RAL-цвета не должно блокировать расчёт и не должно быть обязательным для отправки заявки.
- Логика RAL применяется только к эмали/покраске; не относится к материалу проступей, облицовке, ограждениям и подсветке.
